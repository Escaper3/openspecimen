Steps to follow in order to build your project:

 

1> Create ${username}_${computername}.properties or Edit default.properties under BUILDENGINE_HOME>usersettings

 

2> Add/Edit parameters specific to your project

       Add property project.root.dir to point to your project root

       Add property project.src.dir to point to your project source directory

       Add property project.lib.dir to point to your project lib directory

       Add property project.test.dir to point to your project test directory

       Add property project.uat.dir to point to your project uat directory

 

3> Output directories: By default all reports will be written to the location

   ${project.root.dir}\output

   You can override output.default.dir to point to your output directory

 

4> Set java version: Set the property java.required.version to the java version installed on

   your machine

  

 

5> Open command prompt and move to folder BUILDENGINE_HOME>batch

 

6> Run build.bat. This will compile your code, run all code analyzer tools.

 

6> You can see the output of the build engine under folder specified by property output.default.dir.

 

Note: According to your project requirements, you can override other properties from

        BUILDENGINE_HOME/user settings/default.properties

 

 

 

Following are the tools that are integrated in the build engine:

 

1>  PMD : http://pmd.sourceforge.net/

    PMD scans Java source code and looks for potential problems like:

        o   Empty catch blocks

        o   Empty 'if' statements

        o   Duplicate import statements

        o   Classes which could be Singletons

        o   Short/long variable and method names

        o   Possible bugs - empty try/catch/finally/switch statements

        o   Dead code - unused local variables, parameters and private methods

        o   Suboptimal code - wasteful String/StringBuffer usage

        o   Overcomplicated expressions - unnecessary if statements, for loops that could be while loops

        o   Duplicate code

   

 

2>  CPD : http://pmd.sourceforge.net/cpd.html

    Duplicate code can be hard to find, especially in a large project. But PMD's Copy/Paste Detector (CPD) can find these duplicated pieces of code.

 

3> Cobertura : http://cobertura.sourceforge.net/

    Cobertura is a free Java tool that calculates the percentage of code accessed by junit test cases. It can be used to identify which parts of your Java program are lacking test coverage. Main features of this tool are:

        o   Can be executed from ant or from the command line.

        o   Instruments Java byte code after it has been compiled.

        o   Can generate reports in HTML or XML.

        o   Shows the percentage of lines and branches covered for each class, each package, and for the overall project.

        o   Shows the cyclomatic code complexity of each class, and the average cyclomatic code complexity for each package, and for the overall product.

        o   Can sort HTML results by class name, percent of lines covered, percent of branches covered, etc.

 

 

4> Jdepend : http://clarkware.com/software/JDepend.html

    JDepend traverses Java class file directories and generates design quality metrics for each Java package, including:

        o   Number of Classes and Interfaces

                 The number of concrete and abstract classes (and interfaces) in the package is an indicator of the extensibility of the package.

        o   Afferent Couplings (Ca)

                The number of other packages that depend upon classes within the package is an indicator of the package's responsibility.

        o   Efferent Couplings (Ce)

                The number of other packages that the classes in the package depend upon is an indicator of the package's independence.

        o   Abstractness (A)

                The ratio of the number of abstract classes (and interfaces) in the analyzed package to the total number of classes in the analyzed package. The range for this metric is 0 to 1, with A=0 indicating a completely concrete package and A=1 indicating a completely abstract package.

        o   Instability (I)

                The ratio of efferent coupling (Ce) to total coupling (Ce + Ca) such that I = Ce / (Ce + Ca). This metric is an indicator of the package's resilience to change. The range for this metric is 0 to 1, with I=0 indicating a completely stable package and I=1 indicating a completely instable package.

        o   Package Dependency Cycles

                Package dependency cycles are reported along with the hierarchical paths of packages participating in package dependency cycles.

 

 

5> JCSC : http://jcsc.sourceforge.net/

    JCSC is a powerful tool to check source code against a highly definable coding standard and potential bad code. JCSC can detect bad code in the following areas:

        o   Naming conventions for class, interfaces, fields, and parameter.

        o   Structural layout of the type (class/interface) can be defined. Like where to place fields, either before or after the methods and in which order. The order can be defined through the visibility or by type (instance, class, constant). The same is applicable for methods. Each of those rules is highly customizable.

        o   Placement of white spaces and braces in the code.

        o   The existence of correct JavaDoc can be enforced.

        o   Potential bugs - like empty catch/finally block, switch without default, throwing of type 'Exception', and slow code.

       

 

 

 

6> Macker : http://innig.net/macker/

    Macker is a build-time architectural rule checking utility for Java developers. It's meant to model the architectural ideals programmers always dream up for their projects and then break -- it helps keep code clean and consistent.

    One can tailor a rules file to suit a specific project's structure, or write some general "good practice" rules for the code.

    Macker allows one to specify broad access rules that Java's native access modifiers can't. The current version supports class- and package-level access rules, such as:

        o   Classes in the UI layer may not directly access the data object layer, or use classes in java.sql.

        o   External systems may not access internal implementation classes (suffixed with 'Impl').

        o   One functional module may access another only through its API.

        o   Only classes implementing interfaces in javax.ejb, and certain framework packages, may use the EJB APIs.

 

   

 

7> Spell Check : http://robmayhew.com/antspell/antspelltask.html

    Ant Spell checks the spelling of all files in a directory. Code is broken into single words and compared to a dictionary.

    If the words are not in the dictionary they are written to a file. A project dictionary file can be specified that contains valid words not

    present in the default dictionary

 

   

8> Junit : http://ant.apache.org/manual/OptionalTasks/junit.html

    This task runs all your test cases and UAT's present under test directory. It also generates HTML reports which can help a developer to get test metrics in one glance.

 

 

9> Ant Javadoc Tool: http://ant.apache.org/manual/CoreTasks/javadoc.html

    Generates code documentation using the javadoc tool. The source directory will be recursively scanned for Java source files and it generates HTML Javadoc reports.

 

10 Panopticode : http://www.panopticode.org/  [Work in Progress]

    Panopticode is an open source code analyzer tool with following main features:

         o  Does Unit test code coverage using Emma tool http://emma.sourceforge.net/.

         o  Produces Cyclomatic Complexity and size (NCSS) metrics

         o  Implements CheckStyle

         o  Produces Panopticode coverage and complexity interactive reports in form of treemaps that allow you to see the overall picture and an amazing amount of detail in a single view.
